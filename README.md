# 即投

**BOSS 直聘 AI 自动投递助手** — 一键海投，智能招呼语。

当前版本：**1.3.0** · Chrome 扩展（Manifest V3）

---

## 简介

即投是一款面向 [BOSS 直聘](https://www.zhipin.com) 的浏览器扩展。它帮你在合规前提下，把「搜岗位 → 筛条件 → 写招呼语 → 逐条投递」这条重复链路自动化：

1. **按条件批量采集**匹配岗位（多城市、多期望职位并行）
2. **生成或套用招呼语**，并自动附带简历图片
3. **一键批量投递**，投递完成后汇总结果，支持漏发补投
4. **监听 HR 回复**，识别索要简历的消息并自动发送附件

侧边栏（Side Panel）即主界面，全程在 BOSS 页面旁操作，无需切换窗口。

---

## 功能概览

| 模块 | 说明 |
|------|------|
| 岗位采集 | 按城市、期望职位、行业、薪资、经验、学历、公司规模等筛选；支持岗位/公司名排除词 |
| 智能排序 | 优先公司、方向关键词、按职位偏好、规模标签等多维打分，优先岗位排在前面 |
| 招呼语 | 支持 AI 定制（云端）或**本地固定文案**（当前默认开启本地模式） |
| 简历图片 | 上传 1–2 张清晰简历图，投递时自动跟发；HR 索要简历时可自动回复 |
| 批量投递 | v6 并行架构，多 Worker 后台窗口投递；随机间隔防风控 |
| 测试投递 | 小批量试跑，预览招呼语与实际发送效果后再正式海投 |
| 投递复盘 | 完成后展示成功/失败明细，支持「一键补发」漏投岗位 |
| 去重保护 | 已投递岗位跨批次持久化，避免重复沟通 |
| 数量闸门 | 本地自然日上限 150 条；单批超过 75 条软提示 |

## 安装

### 环境要求

- Google Chrome 或基于 Chromium 的浏览器（Edge、Brave 等）
- 已登录 BOSS 直聘求职者账号
- 开启「开发者模式」以加载 unpacked 扩展

### 加载步骤

1. 克隆或下载本项目到本地目录
2. 打开浏览器扩展管理页：
   - Chrome：`chrome://extensions/`
   - Edge：`edge://extensions/`
3. 开启右上角 **开发者模式**
4. 点击 **加载已解压的扩展程序**，选择项目根目录（含 `manifest.json` 的文件夹）
5. 安装成功后，点击工具栏中的「即投」图标，或在 BOSS 页面右键打开 **侧边栏**

> 更新版本：在扩展管理页点击「重新加载」即可。开发者模式下不会自动推送更新，侧边栏设置里可查看版本提醒。

## 快速上手

### 1. 准备简历

在侧边栏 **设置** 区域或 **选项页**（右键扩展图标 → 选项）上传 1–2 张**清晰**的简历截图（JPG / PNG / WebP）。

### 2. 配置筛选条件

在侧边栏 A 页设置：

- **目标城市**（可多选）
- **期望职位**（可多选，支持自定义职位名）
- **公司行业**、工作区域、薪资、经验、学历等筛选项
- 可选：**岗位名称排除** / **公司名称排除** / **投递优先级** 规则

### 3. 采集岗位

点击 **开始收集**。扩展会自动打开 BOSS 搜索页，按期望职位 × 城市组合采集岗位，进度区展示完整搜索链接供核对。

### 4. 确认并投递

采集完成后进入 B 页，按组查看岗位、预览招呼语，勾选要投递的岗位，点击 **一键发送**。

投递过程中：

- 约需 2–3 分钟（视选中数量而定）
- **请勿手动操作 BOSS 弹窗**
- 可切换到其他标签页，任务在后台继续

### 5. 查看结果

投递结束后进入复盘页，查看每条成功/失败原因；若有漏发，可使用 **一键补发**。

## 测试投递（推荐首次使用）

正式海投前，建议先走一遍测试流程：

1. 在 A 页底部 **测试投递** 区域，设置每个期望职位采集数量（默认 1）
2. 点击 **开始测试投递**（仅使用第一个目标城市）
3. 在结果页预览招呼语，确认无误后点 **测试投递**
4. 到 BOSS 聊天页核对实际发送内容

## 配置说明

### 本地模式（当前默认）

`src/shared/constants.js` 中 `CONFIG.LOCAL_MODE = true` 时：

- 跳过云端 AI 与商业化门禁
- 使用**固定招呼语** + 本地简历图片投递
- 无需配置 API Key

固定招呼语优先级（从高到低）：

1. `FIXED_GREETINGS_BY_CATEGORY` — 按期望职位名映射的专属文案
2. 选项页保存的全局 `fixedGreetingText`
3. `FIXED_GREETING_TEXT` — 代码中的全局默认文案

**按职位配置不同招呼语**示例（编辑 `constants.js`）：

```javascript
FIXED_GREETINGS_BY_CATEGORY: {
  '测试开发': '您好，我有 IoT 测试开发经验……',
  'AI Agent': '您好，我有 Agent 应用开发经验……',
},
```

修改后需在 `chrome://extensions/` 点击扩展的 **重新加载**。

### 选项页

路径：`src/options/options.html`（扩展右键 → 选项）

可配置：

- 简历图片
- 全局固定招呼语
- DeepSeek API Key 与文字简历（本地模式下可忽略）

### 云端 AI 模式（可选）

将 `LOCAL_MODE` 设为 `false` 并配置 API Key 后，扩展会调用阿里云百炼 Qwen 模型，根据简历图片与 JD 生成定制招呼语。此模式涉及会员额度与云端服务，详见扩展内「会员」说明。

## 项目结构

```
jitou-v1.3.0/
├── manifest.json              # 扩展清单（Manifest V3）
├── icons/                     # 扩展图标
├── _locales/                  # 国际化
├── process_position.js        # BOSS 职位树数据处理脚本
└── src/
    ├── background/
    │   └── service-worker.js  # 消息中枢、采集/投递调度、AI 代理
    ├── content/               # 注入 BOSS 页面的脚本
    │   ├── content.js         # Content Script 入口
    │   ├── job-collector.js   # 岗位采集
    │   ├── job-sender.js      # 消息发送
    │   ├── job-clicker.js     # 页面交互（点击立即沟通等）
    │   ├── chat-monitor.js    # HR 消息监听与自动回复
    │   ├── welfare-collector.js
    │   ├── ws-probe.js
    │   ├── selectors.js       # DOM 选择器（须与 SW 消息类型保持同步）
    │   └── ...
    ├── popup/                 # 侧边栏 UI（主界面）
    ├── options/               # 选项页
    ├── db/
    │   └── indexeddb.js       # 简历图片本地存储
    ├── shared/
    │   ├── constants.js       # 全局常量与配置（单一真相源）
    │   ├── device-id.js
    │   ├── diag-logger.js     # 诊断日志
    │   └── error-logger.js
    └── utils/
        └── api.js
```

### 架构要点

- **Popup ↔ Service Worker ↔ Content Script** 三端通过 `MSG` 常量（`constants.js`）通信
- 采集与投递状态持久化在 `chrome.storage.local`（`sw:` / `ui:` 前缀隔离）
- v6 投递采用多 Worker 并行 + 详情页提取 HR 信息的两阶段流水线
- 部分消息类型标注「须镜像 selectors.js」，修改通信协议时需同步两处

## 常见问题

**Q：采集结果为 0 条？**

- 点开进度区展示的 BOSS 搜索链接，确认页面本身是否有岗位
- 若 BOSS 有岗位但扩展为 0，可下载诊断包反馈；若 BOSS 也无岗位，请放宽筛选条件或排除词

**Q：招呼语显示「生成失败」？**

- 本地模式：检查 `constants.js` 或选项页是否配置了有效固定招呼语
- 云端模式：确认简历图片清晰，API Key 有效

**Q：投递中途卡住？**

- 检查是否误触 BOSS 验证码或弹窗
- 在扩展管理页重新加载扩展后，从复盘页尝试补发

**Q：如何反馈问题？**

侧边栏 → 设置（齿轮）→ **问题反馈** → 下载诊断文件（已脱敏），发送给产品负责人。
